# Spec Requirements: AI Tell Removal (humanise)

## Initial Description
We want to extend the functionality of uwotm8 to include additional functions to make text generated by AI processes look less like AI generated text. This means for example replacing long dashes with commas and spaces, avoiding over use of single sentences followed by bullet lists as summary information, using tables and other means of conveying information more proactively, removing special characters, non-breaking spaces, etc. We want to make the text feel as human as possible. We may also need to look at average UK reading age capabilities as a target for grammar, etc. The tool will be used on everything from scientific text to industrial reports to blog posts and podcast content so it needs to be general but it needs to filter out all common AI "tells" to improve the text for the end reader (typically a European business person).

## Requirements Discussion

### First Round Questions

**Q1:** Should this be a new CLI flag on the existing `uwotm8` command or a separate subcommand?
**Answer:** Flag on existing command (e.g. `uwotm8 --humanise`), not a separate subcommand.

**Q2:** Should the humanisation logic live in `convert.py` or a separate module?
**Answer:** Separate module (e.g. `uwotm8/humanise.py`), not integrated into convert.py.

**Q3:** What specific character-level AI tells should be targeted?
**Answer:** The user's initial list plus anything else we can think of. They want to add more in future, so the architecture should be extensible. Current list:
- Em dashes replaced with commas or restructured clauses
- Non-breaking spaces replaced with normal spaces
- Unusual Unicode characters (smart quotes, ellipsis characters, etc.) normalised
- Excessive bold/italic markdown formatting
- Plus any other common AI character-level tells we can identify

**Q4:** How should structural patterns (bullet lists, filler phrases, hedging, etc.) be handled?
**Answer:** BOTH automatic rewriting AND flagging, controlled via a `--rewrite` flag with `all` or specific sub-options for each trait (e.g. bullet lists, filler phrases, hedging, uniform paragraphs, repetitive transitions).

**Q5:** Should reading age be calculated/reported, and should simplifications be suggested?
**Answer:** Both calculate/report AND suggest simplifications. Target should be configurable.

**Q6:** Which file formats should be supported?
**Answer:** .md, .txt, .py, .rst, .html, AND .docx. Plus stdin/stdout.

**Q7:** Should there be an aggressiveness/tier setting?
**Answer:** Yes to tiered approach (minimal, moderate, full).

**Q8:** What is explicitly out of scope?
**Answer:** Non-English text and real-time/watch mode.

### Follow-up Questions

**Follow-up 1:** How should `--rewrite` sub-options be specified? And what is the default behaviour of `--humanise` without `--rewrite`?
**Answer:** Comma-separated list (e.g. `--rewrite bullet-lists,filler-phrases`). `--humanise` defaults to flagging-only; `--rewrite` is an explicit opt-in for automatic rewriting.

**Follow-up 2:** What format should flagging output take?
**Answer:** Combination approach -- terminal report by default, plus `--annotate` flag to add inline comments directly in the file. Additionally, an option to save the terminal report to disk.

**Follow-up 3:** How should .docx support be packaged?
**Answer:** Optional dependency via `pip install uwotm8[docx]`.

**Follow-up 4:** How should reading age be calculated, and how should the target be expressed?
**Answer:** Combination of multiple readability indices (not a single formula). Target is configurable and can be expressed as either an age OR a descriptive level (both options available to the user).

**Follow-up 5:** What format should the AI tells registry use?
**Answer:** Whichever is most maintainable and extensible. A Python dictionary/registry pattern (like the existing `CONVERSION_IGNORE_LIST`) or a class-based registry would both work. The user will use Claude Code to make updates, so developer-friendly is the priority. The spec writer should choose the most appropriate pattern.

**Follow-up 6:** Should `--humanise` automatically run spelling conversion too, or are they independent?
**Answer:** Independent. `--humanise` does NOT automatically run spelling conversion. Users combine flags explicitly (e.g. `uwotm8 --humanise path/` for humanisation only, `uwotm8 --humanise --strict path/` for both).

## Requirements Summary

### Functional Requirements
- Add `--humanise` flag to existing `uwotm8` CLI command (independent of spelling conversion)
- Character-level AI tell removal (em dashes, non-breaking spaces, smart quotes, ellipsis characters, excessive markdown formatting, and more)
- Structural pattern detection with flagging as default behaviour under `--humanise`
- `--rewrite` flag for explicit opt-in to automatic rewriting, accepting comma-separated sub-options (e.g. `--rewrite bullet-lists,filler-phrases`) or `--rewrite all`
- Terminal report output by default showing flagged AI tells
- `--annotate` flag to add inline comments in the file marking flagged tells
- Option to save the terminal report to disk
- Reading age calculation using a combination of multiple readability indices
- Reading age target configurable as either an age (e.g. 14) or a descriptive level (e.g. "general audience")
- Reading age simplification suggestions
- Tiered aggressiveness levels: minimal, moderate, full
- Extensible AI tells registry (Python dictionary or class-based pattern, developer-friendly)
- New module `uwotm8/humanise.py` separate from existing convert.py
- .docx support as optional dependency (`pip install uwotm8[docx]`)
- Support for .md, .txt, .py, .rst, .html, .docx, and stdin/stdout

### Reusability Opportunities
- CLI argument parsing pattern from `convert.py:main()`
- File processing pipeline (`process_paths`, `convert_file`, `_process_file`)
- Streaming support pattern (`convert_stream`)
- Extensible dictionary/registry pattern (`CONVERSION_IGNORE_LIST`)
- Check/dry-run mode pattern
- Output file handling (`_handle_file_with_output`)

### Scope Boundaries
**In Scope:**
- Character-level AI tell normalisation
- Structural pattern detection (flagging by default, rewriting opt-in)
- Terminal report, inline annotation, and report-to-disk output modes
- Reading age calculation (multiple indices), configurable target (age or descriptive level), and simplification suggestions
- Support for .md, .txt, .py, .rst, .html, .docx (optional dep), and stdin/stdout
- Tiered aggressiveness (minimal, moderate, full)
- Extensible tell registry architecture (Python-native, developer-friendly)
- Integration with existing CLI via `--humanise` flag (independent of spelling conversion)
- `--rewrite` with comma-separated sub-options for selective automatic rewriting

**Out of Scope:**
- Non-English text support
- Real-time / watch mode
- Separate subcommand (using flag approach instead)
- Automatic spelling conversion when `--humanise` is used (must be combined explicitly)

### Technical Considerations
- New module `uwotm8/humanise.py` to keep concerns separated
- .docx support via optional dependency (e.g. `python-docx`), installed with `pip install uwotm8[docx]`
- Reading age algorithms need to be suitable for UK English; multiple indices combined
- AI tells registry as a Python-native pattern (dictionary or class-based), chosen for maintainability and extensibility
- `--rewrite` accepts comma-separated values; needs CLI parsing for sub-option validation
- `--annotate` flag adds inline comments; needs format-aware insertion (different for .md vs .html vs .py etc.)
- Report-to-disk option needs a file path argument or sensible default
- `--humanise` is independent of spelling conversion; both can be combined but neither implies the other
- Must integrate cleanly with existing `--strict`, `--check`, `--comments-only` flags
- Structural rewriting is significantly more complex than character-level replacement and may require NLP capabilities
- Existing `--include` flag defaults to `.py .txt .md` -- needs extending for new formats (.rst, .html, .docx)
